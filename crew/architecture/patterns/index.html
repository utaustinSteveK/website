    Design Patterns¶

    This document outlines the key design patterns implemented in the Crew Appraisals Backend system, providing both conceptual explanations and practical code examples from the codebase.

    Repository Pattern¶

    The Repository pattern provides a uniform interface for accessing data, abstracting the underlying data storage mechanisms and enabling easier testing and maintenance.

    Implementation¶

    While not implemented as explicit repository classes, our services follow the repository pattern principles by encapsulating data access logic:

    # Example from src/services/crew_service.py
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select, and_
from src.models import CrewMember, Certificate
from src.schemas.crew import CrewMemberCreate, CrewMemberUpdate

class CrewService:
    """Service acting as repository for crew member operations"""

    def __init__(self, db: AsyncSession):
        self.db = db

    async def get_crew_member(self, crew_id: str) -> CrewMember | None:
        """Get crew member by ID"""
        stmt = select(CrewMember).where(
            and_(
                CrewMember.id == crew_id,
                CrewMember.archived_at.is_(None)  # Exclude soft-deleted
            )
        )
        result = await self.db.execute(stmt)
        return result.scalar_one_or_none()

    async def create_crew_member(self, crew_data: CrewMemberCreate) -> CrewMember:
        """Create new crew member"""
        crew_member = CrewMember(**crew_data.model_dump())
        self.db.add(crew_member)
        await self.db.flush()  # Get ID without committing
        await self.db.refresh(crew_member)
        return crew_member

    async def update_crew_member(
        self,
        crew_id: str,
        crew_data: CrewMemberUpdate
    ) -> CrewMember | None:
        """Update existing crew member"""
        crew_member = await self.get_crew_member(crew_id)
        if not crew_member:
            return None

        for field, value in crew_data.model_dump(exclude_unset=True).items():
            setattr(crew_member, field, value)

        await self.db.flush()
        await self.db.refresh(crew_member)
        return crew_member

    Benefits¶

    • Abstraction: Data access logic is separated from business logic
    • Testability: Easy to mock for unit testing
    • Consistency: Uniform interface across different data operations
    • Maintainability: Changes to data access patterns are centralized

    Unit of Work Pattern¶

    The Unit of Work pattern maintains a list of objects affected by a business transaction and coordinates writing out changes and resolving concurrency problems.

    Implementation¶

    Our implementation uses SQLAlchemy's session management within dependency injection:

    # Example from src/core/database.py
from contextlib import asynccontextmanager
from sqlalchemy.ext.asyncio import AsyncSession

@asynccontextmanager
async def get_db_context() -> AsyncGenerator[AsyncSession, None]:
    """
    Context manager for database session - implements Unit of Work pattern
    """
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()  # Commit all changes as one unit
        except Exception as e:
            await session.rollback()  # Rollback entire unit on error
            logger.error(f"Database context error: {str(e)}")
            raise
        finally:
            await session.close()

# Usage example in complex business operations
async def create_appraisal_with_assignments(
    appraisal_data: AppraisalCreate,
    assignments: List[AssignmentCreate],
    db: AsyncSession
) -> Appraisal:
    """
    Complex operation requiring multiple database operations
    All operations succeed or fail together (Unit of Work)
    """
    # Create appraisal
    appraisal = Appraisal(**appraisal_data.model_dump())
    db.add(appraisal)
    await db.flush()  # Get ID but don't commit yet

    # Create assignments
    for assignment_data in assignments:
        assignment = Assignment(
            appraisal_id=appraisal.id,
            **assignment_data.model_dump()
        )
        db.add(assignment)

    # Send notifications (would be implemented)
    await send_appraisal_notifications(appraisal.id, db)

    # All operations committed together when session closes
    # If any operation fails, everything rolls back
    return appraisal

    FastAPI Integration¶

    # Example from src/api/v1/crew.py
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession
from src.core.database import get_db

@router.post("/", response_model=CrewMemberResponse)
async def create_crew_member(
    crew_data: CrewMemberCreate,
    db: AsyncSession = Depends(get_db)  # Unit of Work scope
):
    """
    The dependency injection provides a session that implements Unit of Work
    All database operations in this endpoint are part of one transaction
    """
    service = CrewService(db)
    crew_member = await service.create_crew_member(crew_data)

    # Additional operations that are part of the same transaction
    await service.assign_initial_certificates(crew_member.id)
    await service.create_audit_log("crew_created", crew_member.id)

    # Transaction commits when the dependency completes successfully
    return crew_member

    Benefits¶

    • ACID Compliance: Ensures data consistency across multiple operations
    • Error Handling: Automatic rollback on failures
    • Performance: Reduces database round trips by batching operations
    • Simplicity: Transparent transaction management

    Dependency Injection Pattern¶

    Dependency Injection promotes loose coupling by removing hard dependencies between components and providing dependencies from external sources.

    Implementation¶

    Our system uses FastAPI's dependency injection system extensively:

    # Example from src/core/deps.py
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer
from jose import jwt, JWTError
from sqlalchemy.ext.asyncio import AsyncSession

from src.core.database import get_db
from src.core.config import settings
from src.models import User
from src.services.auth_service import AuthService

# Security dependency
security = HTTPBearer()

async def get_current_user(
    token: str = Depends(security),
    db: AsyncSession = Depends(get_db)
) -> User:
    """
    Dependency that validates JWT token and returns current user
    Can be injected into any endpoint that requires authentication
    """
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        payload = jwt.decode(
            token.credentials,
            settings.SECRET_KEY,
            algorithms=[settings.ALGORITHM]
        )
        user_id: str = payload.get("sub")
        if user_id is None:
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    auth_service = AuthService(db)
    user = await auth_service.get_user_by_id(user_id)
    if user is None:
        raise credentials_exception

    return user

async def get_admin_user(
    current_user: User = Depends(get_current_user)
) -> User:
    """
    Dependency that ensures the current user has admin privileges
    Builds on top of get_current_user dependency
    """
    if not current_user.is_admin:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="Admin privileges required"
        )
    return current_user

# Service dependencies
def get_crew_service(db: AsyncSession = Depends(get_db)) -> CrewService:
    """Factory function for CrewService dependency"""
    return CrewService(db)

def get_appraisal_service(db: AsyncSession = Depends(get_db)) -> AppraisalService:
    """Factory function for AppraisalService dependency"""
    return AppraisalService(db)

    Usage in Endpoints¶

    # Example from src/api/v1/crew.py
@router.get("/{crew_id}", response_model=CrewMemberResponse)
async def get_crew_member(
    crew_id: str,
    current_user: User = Depends(get_current_user),  # Authentication
    crew_service: CrewService = Depends(get_crew_service)  # Business logic
):
    """
    Dependencies are injected automatically:
    - current_user: Handles authentication and authorization
    - crew_service: Provides business logic operations
    """
    crew_member = await crew_service.get_crew_member(crew_id)
    if not crew_member:
        raise HTTPException(status_code=404, detail="Crew member not found")

    # Authorization check
    if not crew_service.can_access_crew_member(current_user, crew_member):
        raise HTTPException(status_code=403, detail="Access denied")

    return crew_member

@router.put("/{crew_id}", response_model=CrewMemberResponse)
async def update_crew_member(
    crew_id: str,
    crew_data: CrewMemberUpdate,
    admin_user: User = Depends(get_admin_user),  # Admin-only endpoint
    crew_service: CrewService = Depends(get_crew_service)
):
    """
    Uses get_admin_user which automatically includes authentication
    and adds admin privilege validation
    """
    crew_member = await crew_service.update_crew_member(crew_id, crew_data)
    if not crew_member:
        raise HTTPException(status_code=404, detail="Crew member not found")

    return crew_member

    Benefits¶

    • Loose Coupling: Components don't need to know how to create their dependencies
    • Testability: Easy to inject mock dependencies for testing
    • Configurability: Dependencies can be swapped based on environment
    • Reusability: Common dependencies can be shared across endpoints

    Event-Driven Patterns¶

    Event-driven patterns help maintain loose coupling between components and enable audit trails and future integrations.

    Domain Events Implementation¶

    # Example domain event structure
from dataclasses import dataclass
from datetime import datetime
from typing import Any, Dict

@dataclass
class DomainEvent:
    """Base class for domain events"""
    event_type: str
    entity_id: str
    entity_type: str
    user_id: str | None
    timestamp: datetime
    data: Dict[str, Any]

@dataclass
class AppraisalCreatedEvent(DomainEvent):
    """Event raised when an appraisal is created"""
    event_type: str = "appraisal.created"
    entity_type: str = "appraisal"

@dataclass
class CertificateExpiringEvent(DomainEvent):
    """Event raised when a certificate is expiring soon"""
    event_type: str = "certificate.expiring"
    entity_type: str = "certificate"

    Event Publishing in Services¶

    # Example from appraisal service with event publishing
from typing import List
from src.models import Appraisal, AuditLog
from src.schemas.appraisal import AppraisalCreate

class AppraisalService:
    def __init__(self, db: AsyncSession):
        self.db = db
        self._events: List[DomainEvent] = []

    async def create_appraisal(
        self,
        appraisal_data: AppraisalCreate,
        created_by: str
    ) -> Appraisal:
        """Create appraisal and publish domain event"""
        # Create the appraisal
        appraisal = Appraisal(**appraisal_data.model_dump())
        appraisal.created_by = created_by
        self.db.add(appraisal)
        await self.db.flush()

        # Create audit log entry
        audit_entry = AuditLog(
            entity_type="appraisal",
            entity_id=appraisal.id,
            action="created",
            user_id=created_by,
            changes=appraisal_data.model_dump()
        )
        self.db.add(audit_entry)

        # Publish domain event
        event = AppraisalCreatedEvent(
            entity_id=appraisal.id,
            user_id=created_by,
            timestamp=datetime.utcnow(),
            data={
                "appraisal_type": appraisal.appraisal_type,
                "crew_member_id": appraisal.crew_member_id,
                "evaluator_id": appraisal.evaluator_id
            }
        )
        await self._publish_event(event)

        return appraisal

    async def _publish_event(self, event: DomainEvent):
        """Publish domain event (placeholder for future message queue)"""
        # For now, just log the event
        # Future: Publish to message queue (Redis Streams, RabbitMQ, etc.)
        logger.info(f"Domain event published: {event.event_type}", extra={
            "event": event,
            "entity_id": event.entity_id
        })

        # Store events for potential batch processing
        self._events.append(event)

    Event Handlers¶

    # Event handler example for certificate expiry notifications
from datetime import datetime, timedelta

class CertificateEventHandler:
    """Handle certificate-related events"""

    def __init__(self, db: AsyncSession):
        self.db = db

    async def handle_certificate_expiring(self, event: CertificateExpiringEvent):
        """Handle certificate expiring event"""
        certificate = await self.get_certificate(event.entity_id)
        if not certificate:
            return

        # Send notification to crew member
        await self.send_expiry_notification(certificate)

        # Create task for HR team
        await self.create_renewal_task(certificate)

        # Update certificate status
        certificate.status = "expiring_soon"
        await self.db.flush()

    async def handle_appraisal_created(self, event: AppraisalCreatedEvent):
        """Handle new appraisal creation"""
        # Send notification to evaluator
        await self.notify_evaluator(event.data["evaluator_id"])

        # Schedule reminder emails
        await self.schedule_appraisal_reminders(event.entity_id)

        # Update crew member's appraisal status
        await self.update_crew_appraisal_status(
            event.data["crew_member_id"],
            "in_progress"
        )

    Benefits¶

    • Decoupling: Components communicate through events without direct dependencies
    • Audit Trail: All important business events are captured
    • Extensibility: New event handlers can be added without modifying existing code
    • Future Integration: Events can be easily published to external message queues

    CQRS (Command Query Responsibility Segregation)¶

    While not fully implemented, our architecture shows CQRS principles in some areas, particularly in separating read and write operations.

    Read Models (Queries)¶

    # Example read models for complex queries
from pydantic import BaseModel
from typing import List, Optional

class CrewMemberSummary(BaseModel):
    """Read model for crew member list views"""
    id: str
    full_name: str
    rank: str
    vessel_name: Optional[str]
    certificate_status: str
    appraisal_status: str

class AppraisalReport(BaseModel):
    """Read model for appraisal reporting"""
    crew_member_name: str
    evaluator_name: str
    completion_date: Optional[datetime]
    overall_rating: float
    categories: List[Dict[str, Any]]

# Service methods optimized for reads
class CrewQueryService:
    """Specialized service for crew member queries"""

    def __init__(self, db: AsyncSession):
        self.db = db

    async def get_crew_summary(
        self,
        filters: CrewFilters
    ) -> List[CrewMemberSummary]:
        """Optimized query for crew member list view"""
        query = """
        SELECT
            cm.id,
            CONCAT(cm.first_name, ' ', cm.last_name) as full_name,
            cm.current_rank,
            v.name as vessel_name,
            CASE
                WHEN COUNT(c.id) FILTER (WHERE c.expires_at < NOW() + INTERVAL '30 days') > 0
                THEN 'expiring'
                WHEN COUNT(c.id) FILTER (WHERE c.expires_at < NOW()) > 0
                THEN 'expired'
                ELSE 'valid'
            END as certificate_status,
            COALESCE(a.status, 'none') as appraisal_status
        FROM crew_members cm
        LEFT JOIN vessel_assignments va ON cm.id = va.crew_member_id AND va.active = true
        LEFT JOIN vessels v ON va.vessel_id = v.id
        LEFT JOIN certificates c ON cm.id = c.crew_member_id
        LEFT JOIN appraisals a ON cm.id = a.crew_member_id AND a.status = 'in_progress'
        WHERE cm.archived_at IS NULL
        GROUP BY cm.id, v.name, a.status
        ORDER BY cm.last_name, cm.first_name
        """

        result = await self.db.execute(text(query))
        return [CrewMemberSummary(**row._asdict()) for row in result]

    Command Models (Writes)¶

    # Command models for write operations
class CreateAppraisalCommand(BaseModel):
    """Command for creating an appraisal"""
    crew_member_id: str
    evaluator_id: str
    appraisal_type: str
    due_date: datetime
    categories: List[str]

class UpdateCertificateCommand(BaseModel):
    """Command for updating certificate information"""
    certificate_id: str
    issue_date: datetime
    expires_at: datetime
    issuing_authority: str
    document_url: Optional[str]

# Separate command handlers
class AppraisalCommandHandler:
    """Handler for appraisal-related commands"""

    def __init__(self, db: AsyncSession):
        self.db = db

    async def handle_create_appraisal(
        self,
        command: CreateAppraisalCommand
    ) -> str:
        """Handle appraisal creation command"""
        # Validate command
        await self._validate_create_appraisal(command)

        # Create appraisal
        appraisal = Appraisal(
            crew_member_id=command.crew_member_id,
            evaluator_id=command.evaluator_id,
            appraisal_type=command.appraisal_type,
            due_date=command.due_date,
            status="pending"
        )
        self.db.add(appraisal)
        await self.db.flush()

        # Create appraisal categories
        for category_name in command.categories:
            category = AppraisalCategory(
                appraisal_id=appraisal.id,
                category_name=category_name,
                status="pending"
            )
            self.db.add(category)

        # Publish event
        await self._publish_appraisal_created_event(appraisal.id)

        return appraisal.id

    Benefits¶

    • Performance: Optimized read and write operations
    • Scalability: Read and write operations can be scaled independently
    • Complexity Management: Separates simple queries from complex business logic
    • Future Evolution: Enables eventual consistency patterns if needed

    Factory Pattern¶

    The Factory pattern is used to create objects without specifying their exact classes, particularly useful for creating services and configurations.

    Service Factory Implementation¶

    # Example service factory
from typing import Protocol
from src.core.config import settings

class ServiceFactory:
    """Factory for creating service instances"""

    def __init__(self, db: AsyncSession):
        self.db = db

    def create_crew_service(self) -> CrewService:
        """Create crew service with proper configuration"""
        return CrewService(self.db)

    def create_appraisal_service(self) -> AppraisalService:
        """Create appraisal service with proper configuration"""
        return AppraisalService(self.db)

    def create_notification_service(self) -> NotificationService:
        """Create notification service based on configuration"""
        if settings.ENABLE_EMAIL_NOTIFICATIONS:
            return EmailNotificationService(
                smtp_server=settings.SMTP_SERVER,
                smtp_port=settings.SMTP_PORT
            )
        else:
            return LoggingNotificationService()

# Configuration-based factory
class ConfigBasedFactory:
    """Factory that creates services based on configuration"""

    @staticmethod
    def create_cache_service() -> CacheService:
        """Create cache service based on environment"""
        if settings.is_production:
            return RedisCacheService(settings.REDIS_URL)
        elif settings.is_testing:
            return InMemoryCacheService()
        else:
            return DevelopmentCacheService()

    Benefits¶

    • Flexibility: Object creation can be customized based on configuration
    • Testing: Easy to create mock objects for testing
    • Encapsulation: Object creation logic is centralized
    • Configuration Management: Different implementations for different environments

    Conclusion¶

    These design patterns provide a solid foundation for the Crew Appraisals Backend system:

    • Repository Pattern: Provides clean data access abstraction
    • Unit of Work: Ensures data consistency across operations
    • Dependency Injection: Enables loose coupling and testability
    • Event-Driven Patterns: Supports audit trails and future integrations
    • CQRS Principles: Optimizes read and write operations
    • Factory Pattern: Manages object creation and configuration

    The combination of these patterns creates a maintainable, testable, and scalable architecture that can evolve with changing business requirements while maintaining code quality and performance.
