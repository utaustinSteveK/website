    Error Code Reference¶

    This document provides comprehensive information about error codes, HTTP status codes, error response formats, and common error scenarios in the Crewing Management System API.

    Error Response Format¶

    All API errors follow a consistent JSON response format:

    {
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "details": {
      "field": "email",
      "constraint": "Email address already exists"
    },
    "timestamp": "2024-09-26T10:30:00Z",
    "request_id": "req_123456789",
    "path": "/api/v1/crew-members"
  }
}

    Error Response Fields¶

    Field Type Description
    code String Application-specific error code
    message String Human-readable error message
    details Object Additional error context (optional)
    timestamp String ISO 8601 error timestamp
    request_id String Unique request identifier for tracking
    path String API endpoint where error occurred

    HTTP Status Codes¶

    The API uses standard HTTP status codes with the following meanings:

    2xx Success¶

    Code Status Description
    200 OK Request successful
    201 Created Resource created successfully
    202 Accepted Request accepted for processing
    204 No Content Request successful, no content returned

    4xx Client Errors¶

    Code Status Description Common Causes
    400 Bad Request Invalid request format or parameters Malformed JSON, missing required fields
    401 Unauthorized Authentication required or failed Missing/invalid token, expired session
    403 Forbidden Access denied Insufficient permissions, account suspended
    404 Not Found Resource not found Invalid ID, deleted resource
    405 Method Not Allowed HTTP method not supported GET on POST endpoint
    409 Conflict Resource conflict Duplicate unique field, concurrent modification
    422 Unprocessable Entity Validation failed Business rule violation, invalid data
    429 Too Many Requests Rate limit exceeded API throttling, DDoS protection

    5xx Server Errors¶

    Code Status Description Common Causes
    500 Internal Server Error Unexpected server error Database connection, unhandled exception
    502 Bad Gateway Upstream service error Database unavailable, external service down
    503 Service Unavailable Service temporarily unavailable Maintenance mode, overloaded
    504 Gateway Timeout Upstream service timeout Slow database query, external API timeout

    Application Error Codes¶

    Authentication and Authorization (AUTH_*)¶

    Code Description HTTP Status Resolution
    AUTH_INVALID_CREDENTIALS Invalid username or password 401 Check credentials and retry
    AUTH_TOKEN_EXPIRED JWT token has expired 401 Refresh token or re-authenticate
    AUTH_TOKEN_INVALID JWT token is malformed or invalid 401 Clear token and re-authenticate
    AUTH_INSUFFICIENT_PERMISSIONS User lacks required permissions 403 Contact administrator for access
    AUTH_ACCOUNT_SUSPENDED User account is suspended 403 Contact administrator
    AUTH_SESSION_EXPIRED Session has expired 401 Re-authenticate
    AUTH_RATE_LIMITED Too many authentication attempts 429 Wait before retrying

    Validation Errors (VALIDATION_*)¶

    Code Description HTTP Status Resolution
    VALIDATION_REQUIRED_FIELD Required field is missing 422 Provide required field value
    VALIDATION_INVALID_FORMAT Field format is invalid 422 Check field format requirements
    VALIDATION_INVALID_EMAIL Email address format is invalid 422 Provide valid email address
    VALIDATION_INVALID_DATE Date format or value is invalid 422 Use ISO 8601 date format
    VALIDATION_INVALID_ENUM Enum value is not allowed 422 Use one of the allowed values
    VALIDATION_STRING_TOO_LONG String exceeds maximum length 422 Shorten the field value
    VALIDATION_STRING_TOO_SHORT String below minimum length 422 Lengthen the field value
    VALIDATION_NUMBER_OUT_OF_RANGE Number outside allowed range 422 Use value within range
    VALIDATION_DUPLICATE_VALUE Value must be unique 422 Use a different value
    VALIDATION_INVALID_REFERENCE Referenced entity doesn't exist 422 Use valid entity ID

    Resource Errors (RESOURCE_*)¶

    Code Description HTTP Status Resolution
    RESOURCE_NOT_FOUND Requested resource not found 404 Check resource ID
    RESOURCE_ALREADY_EXISTS Resource with unique field exists 409 Use different unique values
    RESOURCE_CONFLICT Resource state conflict 409 Refresh and retry
    RESOURCE_LOCKED Resource is locked by another process 409 Wait and retry
    RESOURCE_DELETED Resource has been soft deleted 410 Restore resource if needed

    Business Logic Errors (BUSINESS_*)¶

    Code Description HTTP Status Resolution
    BUSINESS_INVALID_STATE_TRANSITION Invalid workflow state change 422 Check allowed transitions
    BUSINESS_CERTIFICATE_EXPIRED Certificate has expired 422 Renew certificate
    BUSINESS_CREW_LIMIT_EXCEEDED Vessel crew limit exceeded 422 Remove crew or increase limit
    BUSINESS_INVALID_DATE_RANGE Date range is invalid 422 Ensure start < end dates
    BUSINESS_OVERLAPPING_ASSIGNMENT Crew assignment overlap 409 Resolve assignment conflicts
    BUSINESS_INSUFFICIENT_RANK User rank insufficient for action 403 Assign higher rank
    BUSINESS_APPRAISAL_LOCKED Appraisal is locked 409 Complete workflow to unlock
    BUSINESS_VESSEL_UNAVAILABLE Vessel not available for assignment 422 Check vessel status

    File and Document Errors (FILE_*)¶

    Code Description HTTP Status Resolution
    FILE_NOT_FOUND Requested file not found 404 Check file path
    FILE_TOO_LARGE File exceeds size limit 413 Reduce file size
    FILE_INVALID_TYPE File type not allowed 422 Use supported file type
    FILE_UPLOAD_FAILED File upload failed 500 Retry upload
    FILE_VIRUS_DETECTED Virus detected in file 422 Scan file and retry
    FILE_CORRUPTED File is corrupted 422 Upload valid file

    External Service Errors (EXTERNAL_*)¶

    Code Description HTTP Status Resolution
    EXTERNAL_SERVICE_UNAVAILABLE External service unavailable 503 Retry later
    EXTERNAL_API_ERROR External API returned error 502 Check external service status
    EXTERNAL_TIMEOUT External service timeout 504 Retry with longer timeout
    EXTERNAL_RATE_LIMITED External service rate limited 429 Reduce request frequency

    System Errors (SYSTEM_*)¶

    Code Description HTTP Status Resolution
    SYSTEM_MAINTENANCE System under maintenance 503 Wait for maintenance completion
    SYSTEM_DATABASE_ERROR Database connection error 500 Contact system administrator
    SYSTEM_CONFIGURATION_ERROR System misconfiguration 500 Contact system administrator
    SYSTEM_OVERLOADED System overloaded 503 Retry later or contact support

    Error Handling Best Practices¶

    Client-Side Handling¶

    1. Always check HTTP status codes before processing responses
    2. Parse error responses to get detailed error information
    3. Display user-friendly messages based on error codes
    4. Implement retry logic for transient errors (5xx, 429)
    5. Log errors with request IDs for debugging

    Common Retry Scenarios¶

    • 429 Too Many Requests: Implement exponential backoff
    • 500 Internal Server Error: Retry up to 3 times
    • 502 Bad Gateway: Retry with increasing delays
    • 503 Service Unavailable: Check maintenance status
    • 504 Gateway Timeout: Increase timeout and retry

    Error Prevention¶

    1. Validate input before sending requests
    2. Handle authentication expiry proactively
    3. Check resource existence before operations
    4. Respect rate limits in client applications
    5. Use proper HTTP methods for operations

    Request ID Usage¶

    Every error response includes a request_id field. Use this ID when:

    • Reporting bugs or issues to support
    • Correlating errors across multiple services
    • Debugging complex workflows
    • Tracking request flows in logs

    Monitoring and Alerting¶

    Set up monitoring for:

    • High error rates (> 5% 4xx, > 1% 5xx)
    • Authentication failures (potential security issues)
    • Resource conflicts (business logic problems)
    • External service errors (dependency issues)
    • File upload failures (storage problems)

    Support and Troubleshooting¶

    Getting Help¶

    When reporting errors, provide:

    1. Request ID from error response
    2. HTTP status code and error code
    3. Request details (method, path, payload)
    4. Timestamp of the error
    5. User context (role, permissions)

    Common Solutions¶

    • Authentication errors: Check token validity and refresh if needed
    • Permission errors: Verify user role and required permissions
    • Validation errors: Review API documentation for field requirements
    • Business logic errors: Check entity states and business rules
    • Rate limiting: Implement proper backoff strategies

    For additional support, consult the FAQ or contact the development team with the error details above.
