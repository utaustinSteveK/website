    Core Entity Documentation¶

    This document provides detailed documentation for each core entity in the Crewing Management System, including their attributes, relationships, validation rules, and business logic.

    User Model¶

    The User model handles authentication, authorization, and user management within the system.

    Purpose¶

    • System authentication and session management
    • Role-based access control and permissions
    • User profile and preference management
    • Audit trail for user activities

    Attributes¶

    Field Type Required Constraints Description
    id UUID Yes Primary Key Unique identifier
    username String(50) Yes Unique, Index Login username
    email String(255) Yes Unique, Index Email address
    password_hash String(255) Yes - Bcrypt hashed password
    first_name String(100) Yes - User's first name
    last_name String(100) Yes - User's last name
    phone String(20) No - Contact phone number
    role UserRole Yes Enum User's system role
    is_active Boolean Yes Default: True Account active status
    is_superuser Boolean Yes Default: False Superuser privileges
    preferences JSON No Default: {} User preferences
    created_at DateTime Yes Auto Record creation timestamp
    updated_at DateTime Yes Auto Last update timestamp

    User Roles¶

    Role Permissions Description
    SUPER_ADMIN All permissions System administrator
    ADMIN Crew, appraisal, vessel, form, report management Fleet administrator
    CREW_MANAGER Crew and appraisal management HR and crew operations
    RECRUITMENT_OFFICER Crew recruitment and certificates Hiring and compliance
    VESSEL_CAPTAIN Crew appraisals on assigned vessel Ship captain
    DEPARTMENT_HEAD Department crew appraisals Department manager
    CREW_MEMBER Personal profile and appraisal view Individual crew member
    VIEWER Read-only access to reports Observer role

    Relationships¶

    • One-to-One: crew_member (optional)
    • One-to-Many: sessions, audit_logs
    • Evaluator: appraisals (as evaluator or reviewer)

    Methods¶

    • full_name: Returns formatted full name
    • set_password(password): Hash and store password
    • verify_password(password): Verify password against hash
    • has_permission(permission): Check role-based permissions

    Validation Rules¶

    • Username must be unique and 3-50 characters
    • Email must be valid format and unique
    • Password must meet complexity requirements (handled externally)
    • Phone number format validation (optional)

    CrewMember Model¶

    The CrewMember model manages comprehensive personnel records for maritime crew members.

    Purpose¶

    • Store detailed crew member information
    • Track employment status and history
    • Manage emergency contacts and personal details
    • Support regulatory compliance requirements

    Attributes¶

    Personal Information¶

    Field Type Required Constraints Description
    id UUID Yes Primary Key Unique identifier
    first_name String(100) Yes - First name
    last_name String(100) Yes - Last name
    middle_name String(100) No - Middle name
    email String(255) Yes Unique Email address
    phone String(50) No - Primary phone
    phone_secondary String(50) No - Secondary phone

    Professional Information¶

    Field Type Required Constraints Description
    employee_id String(50) Yes Unique Company employee ID
    rank String(100) Yes - Maritime rank/position
    department Department Yes Enum Ship department
    status CrewStatus Yes Default: AVAILABLE Employment status

    Personal Details¶

    Field Type Required Constraints Description
    date_of_birth Date No - Birth date
    nationality String(100) No - Nationality
    passport_number String(50) No - Passport number
    passport_expiry Date No - Passport expiration
    seaman_book_number String(50) No - Seaman's book number

    Employment Details¶

    Field Type Required Constraints Description
    date_joined Date Yes - Employment start date
    date_left Date No - Employment end date
    contract_type String(50) No - Contract type
    contract_duration_months Integer No - Contract duration

    Contact Information¶

    Field Type Required Constraints Description
    address_line1 String(255) No - Primary address
    address_line2 String(255) No - Secondary address
    city String(100) No - City
    state_province String(100) No - State or province
    postal_code String(20) No - Postal code
    country String(100) No - Country

    Emergency Contact¶

    Field Type Required Constraints Description
    emergency_contact_name String(200) No - Contact name
    emergency_contact_relationship String(50) No - Relationship
    emergency_contact_phone String(50) No - Contact phone
    emergency_contact_email String(255) No - Contact email

    Additional Information¶

    Field Type Required Constraints Description
    notes Text No - General notes
    metadata_ JSON No - Additional metadata
    photo_url String(500) No - Profile photo URL
    user_id UUID No Foreign Key Linked user account

    Enumerations¶

    CrewStatus¶

    • ACTIVE: Currently employed
    • ON_LEAVE: On approved leave
    • ON_VESSEL: Currently deployed
    • AVAILABLE: Available for assignment
    • INACTIVE: Not currently active
    • TRAINING: In training program
    • MEDICAL: Medical leave

    Department¶

    • DECK: Deck department
    • ENGINE: Engine department
    • CATERING: Catering/galley
    • ELECTRICAL: Electrical
    • RADIO: Radio/communications

    Relationships¶

    • Many-to-One: user (optional)
    • One-to-Many: certificates, vessel_assignments, appraisals

    Methods¶

    • full_name: Returns formatted full name with middle name
    • current_vessel_assignment: Gets active vessel assignment
    • has_valid_certificates(): Checks certificate validity

    Validation Rules¶

    • Employee ID must be unique
    • Email must be unique and valid format
    • Date of birth must be reasonable (18+ years old)
    • Passport expiry must be future date if provided
    • Employment dates must be logical (joined <= left)

    Appraisal Model¶

    The Appraisal model manages crew performance evaluations with workflow support and detailed scoring.

    Purpose¶

    • Track crew member performance evaluations
    • Manage appraisal workflow and approvals
    • Store detailed feedback and recommendations
    • Support compliance and career development

    Attributes¶

    Appraisal Information¶

    Field Type Required Constraints Description
    id UUID Yes Primary Key Unique identifier
    appraisal_number String(50) Yes Unique Business identifier
    type AppraisalType Yes Enum Evaluation type
    status AppraisalStatus Yes Default: DRAFT Workflow status

    Period and Scoring¶

    Field Type Required Constraints Description
    period_start Date Yes - Evaluation period start
    period_end Date Yes - Evaluation period end
    overall_rating Float No 0-5 scale Overall performance rating
    overall_score Float No - Calculated total score
    max_score Float No - Maximum possible score

    Feedback and Comments¶

    Field Type Required Constraints Description
    evaluator_comments Text No - Evaluator feedback
    crew_comments Text No - Crew member response
    improvement_areas Text No - Areas for improvement
    achievements Text No - Notable achievements

    Recommendations¶

    Field Type Required Constraints Description
    recommendation String(255) No - Overall recommendation
    recommended_for_promotion Boolean Yes Default: False Promotion recommendation
    recommended_training JSON Array No - Training recommendations

    Conduct and Discipline¶

    Field Type Required Constraints Description
    conduct_rating Integer No 1-5 scale Conduct rating
    discipline_issues Text No - Disciplinary notes
    commendations Text No - Commendations

    Contract and Rehire¶

    Field Type Required Constraints Description
    recommend_rehire Boolean No - Rehire recommendation
    next_contract_duration_months Integer No - Suggested contract length
    wage_increase_percentage Float No - Wage increase recommendation

    Timestamps¶

    Field Type Required Constraints Description
    evaluation_date DateTime Yes - Evaluation completed date
    reviewed_date DateTime No - Review completion date
    completed_date DateTime No - Final completion date

    Acknowledgment¶

    Field Type Required Constraints Description
    crew_acknowledged Boolean Yes Default: False Crew acknowledgment
    crew_acknowledged_date DateTime No - Acknowledgment date
    crew_signature Text No - Digital signature

    References¶

    Field Type Required Constraints Description
    crew_member_id UUID Yes Foreign Key Evaluated crew member
    vessel_id UUID No Foreign Key Vessel where evaluated
    evaluator_id UUID Yes Foreign Key Evaluating user
    reviewer_id UUID No Foreign Key Reviewing user

    Additional Data¶

    Field Type Required Constraints Description
    attachments JSON Array No - Supporting documents
    metadata_ JSON No - Additional metadata

    Enumerations¶

    AppraisalStatus¶

    • DRAFT: Initial draft state
    • IN_PROGRESS: Currently being completed
    • PENDING_REVIEW: Awaiting review
    • REVIEWED: Review completed
    • COMPLETED: Final completed state
    • DISPUTED: Under dispute

    AppraisalType¶

    • REGULAR: Regular performance review
    • PROBATION: Probationary evaluation
    • PROMOTION: Promotion assessment
    • INCIDENT: Incident-related review
    • END_OF_CONTRACT: Contract completion review
    • SPECIAL: Special circumstances

    Relationships¶

    • Many-to-One: crew_member, vessel, evaluator, reviewer
    • One-to-Many: categories (AppraisalCategory)

    Methods¶

    • calculate_overall_score(): Calculates score from categories
    • is_overdue: Property checking if past due
    • rating_description: Property returning rating description

    Validation Rules¶

    • Period end must be after period start
    • Overall rating must be 0-5 if provided
    • Evaluation date must be within evaluation period
    • Crew member must exist
    • Evaluator must have appropriate permissions

    Vessel Model¶

    The Vessel model manages fleet information, specifications, and operational status.

    Purpose¶

    • Store comprehensive vessel information
    • Track operational status and capabilities
    • Manage crew requirements and assignments
    • Support compliance and inspection tracking

    Attributes¶

    Basic Information¶

    Field Type Required Constraints Description
    id UUID Yes Primary Key Unique identifier
    name String(200) Yes - Vessel name
    imo_number String(20) Yes Unique IMO identification
    mmsi String(20) No Unique MMSI number
    call_sign String(20) No - Radio call sign

    Type and Status¶

    Field Type Required Constraints Description
    type VesselType Yes Enum Vessel type
    status VesselStatus Yes Default: ACTIVE Operational status

    Registration¶

    Field Type Required Constraints Description
    flag_state String(100) Yes - Flag state
    port_of_registry String(100) No - Registry port
    registration_number String(50) No - Registration number

    Technical Specifications¶

    Field Type Required Constraints Description
    gross_tonnage Float No Positive Gross tonnage
    net_tonnage Float No Positive Net tonnage
    deadweight_tonnage Float No Positive Deadweight tonnage
    length_overall Float No Positive Length overall (meters)
    beam Float No Positive Beam (meters)
    draft Float No Positive Draft (meters)

    Build Information¶

    Field Type Required Constraints Description
    year_built Integer No Reasonable year Construction year
    builder String(200) No - Shipbuilder name
    class_society String(100) No - Classification society
    class_notation String(200) No - Class notation

    Crew Requirements¶

    Field Type Required Constraints Description
    minimum_crew Integer Yes Positive Minimum crew required
    maximum_crew Integer Yes >= minimum_crew Maximum crew capacity
    current_crew_count Integer Yes Default: 0 Current crew count

    Management¶

    Field Type Required Constraints Description
    owner String(200) No - Vessel owner
    operator String(200) No - Operating company
    manager String(200) No - Management company

    Operations¶

    Field Type Required Constraints Description
    last_port String(200) No - Last port of call
    next_port String(200) No - Next port destination
    eta Date No - Estimated time of arrival

    Compliance¶

    Field Type Required Constraints Description
    last_inspection_date Date No - Last inspection
    next_inspection_date Date No - Next inspection due
    last_drydock_date Date No - Last drydock
    next_drydock_date Date No - Next drydock due

    Additional Information¶

    Field Type Required Constraints Description
    notes String(1000) No - General notes
    metadata_ JSON No - Additional metadata
    photo_url String(500) No - Vessel photo URL

    Enumerations¶

    VesselType¶

    • BULK_CARRIER: Bulk carrier vessel
    • CONTAINER: Container ship
    • TANKER: Tanker vessel
    • GENERAL_CARGO: General cargo ship
    • PASSENGER: Passenger vessel
    • RORO: Roll-on/roll-off
    • CRUISE: Cruise ship
    • OFFSHORE: Offshore vessel
    • TUG: Tugboat
    • SUPPLY: Supply vessel
    • OTHER: Other vessel type

    VesselStatus¶

    • ACTIVE: Currently active
    • LAID_UP: Laid up/inactive
    • DRY_DOCK: In dry dock
    • DECOMMISSIONED: Decommissioned
    • UNDER_CONSTRUCTION: Under construction

    Relationships¶

    • One-to-Many: assignments (VesselAssignment), appraisals

    Methods¶

    • is_fully_crewed: Property checking minimum crew
    • available_positions: Property calculating available positions
    • update_crew_count(): Updates crew count from assignments

    Validation Rules¶

    • IMO number must be valid format and unique
    • MMSI must be unique if provided
    • Maximum crew must be >= minimum crew
    • Technical specifications must be positive numbers
    • Year built must be reasonable
    • Crew counts must be non-negative

    Certificate Model¶

    The Certificate model manages crew member certifications, licenses, and compliance documents.

    Purpose¶

    • Track crew member certifications and licenses
    • Monitor expiry dates and renewal requirements
    • Ensure regulatory compliance
    • Support verification and audit processes

    Attributes¶

    Certificate Information¶

    Field Type Required Constraints Description
    id UUID Yes Primary Key Unique identifier
    type String(100) Yes - Certificate type
    number String(100) Yes Unique Certificate number
    name String(255) Yes - Certificate name

    Validity¶

    Field Type Required Constraints Description
    issue_date Date Yes - Issue date
    expiry_date Date No - Expiry date
    is_lifetime Boolean Yes Default: False Lifetime validity

    Issuing Authority¶

    Field Type Required Constraints Description
    issued_by String(255) Yes - Issuing authority
    issuing_country String(100) No - Issuing country

    Status¶

    Field Type Required Constraints Description
    is_required Boolean Yes Default: True Required for employment
    is_verified Boolean Yes Default: False Verification status
    verified_date DateTime No - Verification date
    verified_by String(255) No - Verified by whom

    Document Storage¶

    Field Type Required Constraints Description
    document_url String(500) No - Document file URL
    document_hash String(255) No - Document hash

    Additional Information¶

    Field Type Required Constraints Description
    notes Text No - Additional notes
    metadata_ JSON No - Additional metadata
    renewal_alert_days Integer Yes Default: 90 Alert days before expiry
    crew_member_id UUID Yes Foreign Key Owning crew member

    Common Certificate Types¶

    • STCW Certificates: Standards of Training, Certification and Watchkeeping
    • Medical Certificates: Medical fitness certificates
    • Flag State Endorsements: National endorsements
    • Passport: Identity document
    • Seaman's Book: Seaman's identity document
    • Visas: Country-specific entry permits
    • Vaccination Certificates: Health compliance documents

    Relationships¶

    • Many-to-One: crew_member

    Methods¶

    • is_valid(): Checks current validity status
    • is_expiring_soon(days): Checks if expiring within specified days
    • is_expired(): Checks if already expired
    • status: Property returning current status string

    Validation Rules¶

    • Certificate number must be unique
    • Issue date must be in the past
    • Expiry date must be after issue date if provided
    • Renewal alert days must be positive
    • Document hash must match stored document

    Status Values¶

    • valid: Currently valid certificate
    • expiring_soon: Valid but expiring within alert period
    • expired: Past expiry date
    • unknown: Status cannot be determined

    Summary¶

    These core entities form the foundation of the Crewing Management System, providing comprehensive data management for:

    • User authentication and authorization
    • Personnel and crew management
    • Performance evaluation and tracking
    • Fleet and vessel management
    • Compliance and certification tracking

    Each entity is designed with maritime industry requirements in mind, supporting regulatory compliance, operational efficiency, and data integrity throughout the crew management lifecycle.
